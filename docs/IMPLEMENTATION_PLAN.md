## 0. Резюме первого релиза

| Параметр | Решение |
|----------|---------|
| Тип продукта | Web-приложение (SaaS-архитектура) |
| Развёртывание v1 | **Локально** (Docker на ПК); позже — VPS в интернете без смены архитектуры |
| Пользователь | Один предприниматель = владелец = админ; **ролей нет** |
| Языки | Русский, английский, иврит (RTL) |
| Налоговый режим v1 | Только **עוסק פטור**; задел на עוסק מורשה / חברה (בע״מ) |
| Ниша-эталон | Контроллеры дверей, замки, светофоры, комплектующие, монтаж; модель расширяема |
| Шаблоны PDF | Предоставляет владелец; разработка — генерация и подстановка данных |
| Trial / подписка | Логика в коде; **проверка отключена** на этапе отладки (`SUBSCRIPTION_ENFORCEMENT=false`) |
| Код | Не начинать до команды владельца «старт разработки» |

### Scope v1 (включено)

- Регистрация, профиль бизнеса (פטור), пароль
- Клиенты, каталог с артикулами (префикс + номер), BOM, склад
- Торговля **и** услуги в одном заказе
- Цепочка: **הצעת מחיר → заказ → выполнение → חשבון חיוב → оплата → קבלה → закрыт**
- Скидки: разовые и постоянные по клиенту
- Список документов по месяцам (UI-референс), статусы פתוח / סגור
- i18n + RTL, optimistic locking, автосохранение черновиков

### Scope v1 (исключено — позже)

- **חשבונית מס**, מע״מ, עוסק מורשה / בע״מ (режим в коде, UI скрыт)
- Возврат товара, זיכוי
- Несколько пользователей и роли
- API רשות המסים, WhatsApp, мобильное native-приложение
- Банковский перевод для оплаты **подписки** (только карта на проде)

## 2. Артикулы (SKU): префикс + номер

**Формат:** `ПРЕФИКС-NNNNN` (5 цифр, автонумерация **внутри каждого префикса**).

| Префикс | Тип | Пример | Описание |
|---------|-----|--------|----------|
| **FG** | Finished Good | `FG-00001` | Готовое изделие (замок, контроллер, светофор) |
| **CP** | Component Part | `CP-00001` | Комплектующая, закупка у поставщика |
| **SV** | Service | `SV-00001` | Услуга (монтаж, настройка, выезд) |
| **CH** | Charge | `CH-00001` | Прочее (доставка, упаковка) |
| **BD** | Bundle | `BD-00001` | Опционально: явная метка «составной рецепт» (иначе FG + BOM) |
| **SP** | Spare | `SP-00001` | Опционально: расходники / ЗИП |

**Правила:**

- При создании позиции система предлагает префикс по типу и следующий свободный номер.
- Артикул не меняется после первого складского движения; допускается только деактивация.
- Составной товар: артикул **FG-** (или **BD-**) + строки BOM с артикулами **CP-**.
- В PDF клиенту — одна строка на готовое изделие, если галочка «скрыть состав» включена.

### 2.1 Нумерация документов (отдельно от товаров)

| Префикс | Документ | Пример |
|---------|----------|--------|
| **Q** | הצעת מחיר | `Q-10001` |
| **O** | Заказ (внутренний) | `O-40001` |
| **H** | חשבון חיוב | `H-20001` |
| **R** | קבלה | `R-30001` |
| **M** | חשבונית מס | Зарезервировано для `MURSHE` (не v1) |

Выданные номера **не удаляются** — только storno / void. Отдельная серия на налоговый год (настройка).

## 3. Цепочка заказа (עוסק פטור)

```
הצעת מחיר  →  (заказ / форма — опционально в БД)  →  выполнение
  →  חשבון חיוב  →  ожидание оплаты  →  קבלה  →  заказ закрыт (סגור)
```

| # | Этап | Документ / действие | Статус заказа (логика) |
|---|------|---------------------|-------------------------|
| 1 | Предложение | **הצעת מחיר** (Q-) | Черновик → отправлено |
| 2 | Заказ | Форма клиента (опционально), внутренний **O-** | Принят |
| 3 | Выполнение | Услуга +/или списание со склада (FG + BOM) | В работе |
| 4 | Счёт | **חשבון חיוב** (H-) | Счёт выставлен → ждёт оплаты |
| 5 | Оплата | Фиксация оплаты | Оплачено |
| 6 | Квитанция | **קבלה** (R-) | Закрыт (סגור) |

**UI-референс:** список документов **по месяцам** (מאי 2026, אפריל 2026…), колонки: статус, номер, тип, клиент, описание, даты, сумма ₪, действия (email, создать следующий документ).

**Статусы документа в списке (как на референсе):**

| Бейдж | Иврит | HEX | Когда |
|-------|-------|-----|-------|
| Открыт | פתוח | `#EA580C` | Документ/цепочка не завершена |
| Закрыт | סגור | `#0D9488` | После קבלה, заказ выполнен |

**Дополнительные бейджи этапов заказа:**

| Этап | HEX |
|------|-----|
| Черновик | `#94A3B8` |
| Отправлено / ждём клиента | `#3B82F6` |
| В работе | `#0D9488` |
| Счёт выставлен | `#6366F1` |
| Ждёт оплаты | `#D97706` |
| Оплачено | `#059669` |
| Выполнен | `#047857` |
| Просрочка (только бейдж) | `#DC2626` |

## 4. Каталог, BOM, склад

### 4.1 Типы позиций

- **CP** — комплектующие на складе
- **FG** — готовая продукция; при отгрузке списывается FG и автоматически компоненты по BOM × количество
- **SV** — услуги без склада
- **CH** — прочие строки без склада

### 4.2 Составной товар (BOM)

- Компоненты — только **CP-** (или FG как подсборка — редко, v1: CP)
- Галочки при создании FG/BD:
  - [ ] Показывать состав в הצעת מחיר
  - [ ] Показывать состав в חשבון חיוב
- В документе клиенту: одна строка, если состав скрыт
- На складе: −N готовых FG + −(N × BOM) по каждому CP

### 4.3 Склад v1

- **Два склада:** комплектующие (CP/SP), готовые изделия (FG/BD)
- Остатки, журнал движений (приход / расход)
- Списание при **выполнении** заказа (не при הצעת מחיר) — см. `docs/STOCK_POLICY.md`
- **Возврат** — не в v1 (таблицу `Return` можно не создавать до v1.1)

### 4.4 Скидки (в v1)

| Тип | Поведение |
|-----|-----------|
| Разовая | % или сумма на строку / на весь документ |
| Постоянная | В карточке клиента; подставляется в quote и חשבון חיוב, можно переопределить |

## 5. Подписка и trial

### 5.1 Бизнес-логика (код всегда присутствует)

| Параметр | Значение |
|----------|----------|
| Trial | 30 дней с `registered_at` |
| День 31 без оплаты | **Вход разрешён** |
| Запись | **Заблокирована** (новые заказы, документы, клиенты, товары) |
| Чтение | Только документы и данные, созданные **до** окончания trial |
| UI | Экран: «Пробный период закончился — оплатите подписку» |
| Оплата подписки | **Только карта** (Tranzila / Cardcom / Stripe — выбор при продакшене) |

### 5.2 Режим отладки v1

```env
SUBSCRIPTION_ENFORCEMENT=false   # локально и на этапе первого релиза
```

- Middleware проверки подписки **реализован**, но при `false` пропускает все запросы.
- На продакшене: `SUBSCRIPTION_ENFORCEMENT=true` + webhook платежа.

## 10. Техническая архитектура

| Слой | Решение |
|------|---------|
| Frontend | SPA (React или Vue) + TypeScript + i18n + RTL |
| Backend | API (.NET 8 или Node — выбор при старте кода) |
| БД | PostgreSQL |
| Файлы | Локальная папка / volume в v1; S3 при переносе на VPS |
| Auth | JWT или session, bcrypt |
| Multi-tenant | `tenant_id` во всех бизнес-таблицах |
| Контейнеры | Docker Compose: `api`, `db`, `nginx` (опционально на локали) |

### 10.1 Развёртывание

**Фаза 1 — локально (v1):**

1. Docker Compose на ПК разработчика
2. `http://localhost` + PostgreSQL
3. `SUBSCRIPTION_ENFORCEMENT=false`
4. Бэкап: pg_dump вручную или скрипт

**Фаза 2 — продакшен (после отладки):**

1. VPS (Hetzner / DigitalOcean / Azure IL)
2. Тот же Docker Compose + домен + Let's Encrypt
3. Перенос: dump → restore БД
4. `SUBSCRIPTION_ENFORCEMENT=true` + платёж картой
5. Ежедневные бэкапы

### 10.2 Обновления без потери данных

- Optimistic locking (`version` на сущностях)
- Автосохранение черновиков (quote, заказ, H-)
- Rolling deploy; миграции БД additive
- При конфликте версии — toast, форма не закрывается
